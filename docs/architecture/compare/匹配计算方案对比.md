# 匹配计算方案对比分析

## 背景

匹配计算是智能招聘系统的核心功能，负责将用户技能与职位要求进行匹配，生成匹配度评分和推荐理由。

## 各方案详细分析

### 方案1：纯规则匹配

**原理**：使用预定义规则和权重进行匹配计算。

```java
public MatchResult calculateByRules(User user, Job job) {
    // 技能匹配（集合交集）
    Set<String> userSkills = user.getSkills();
    Set<String> requiredSkills = job.getRequiredSkills();
    double skillScore = Sets.intersection(userSkills, requiredSkills).size()
                       * 1.0 / requiredSkills.size();

    // 经验匹配
    int expScore = user.getExperienceYears() >= job.getMinExperience() ? 100 : 0;

    // 城市匹配
    int cityScore = user.getTargetCity().equals(job.getCity()) ? 100 : 0;

    // 加权求和
    double overall = skillScore * 0.5 + expScore * 0.2 + cityScore * 0.15 + ...;

    return new MatchResult(overall, skillScore, expScore, cityScore);
}
```

| 指标 | 评分 | 说明 |
|------|------|------|
| 精准度 | 50% | 无法处理语义相似（如Spring Boot≈Spring） |
| 复杂度 | ⭐ | 纯Java代码，无外部依赖 |
| 性能 | 极快 | 内存计算，毫秒级 |
| 成本 | 低 | 无API费用 |

**问题**：
- 无法识别同义词
- 无法生成自然语言匹配理由
- 规则难以覆盖所有情况

### 方案3：大模型+规则混合

**原理**：大模型处理语义部分（技能归一化、理由生成），规则处理精确部分（城市、学历完全匹配）。

```java
public MatchResult calculateHybrid(User user, Job job) {
    // 1. 规则计算精确匹配（城市、学历、薪资范围）
    int cityScore = user.getTargetCity().equals(job.getCity()) ? 100 : 0;
    int eduScore = compareEducation(user.getEducation(), job.getEducation());
    int salaryScore = calculateSalaryScore(user.getExpectedSalary(), job.getSalaryRange());

    // 2. 大模型处理技能匹配（语义归一化）
    String prompt = String.format("""
        将以下技能归一化到标准技能标签：
        用户技能：%s
        职位要求技能：%s

        标准技能标签包括：Java, Spring, MySQL, PostgreSQL, Redis, Kafka, Vue, React, Git, Docker...

        返回JSON：
        {
          "normalized_user_skills": ["Java", "Spring", "MySQL"],
          "normalized_job_skills": ["Java", "Spring", "Redis"],
          "skill_score": 匹配分(0-100),
          "reason": "简短匹配原因"
        }
        """, user.getSkills(), job.getRequiredSkills());

    SkillMatchResult skillResult = llm.extractAndMatch(prompt);

    // 3. 加权计算最终得分
    double overall = skillResult.getScore() * 0.5
                  + experienceScore * 0.2
                  + cityScore * 0.15
                  + eduScore * 0.1
                  + salaryScore * 0.05;

    return new MatchResult(overall, skillResult, cityScore, eduScore, salaryScore);
}
```

| 指标 | 评分 | 说明 |
|------|------|------|
| 精准度 | 85% | 规则保证精确，大模型处理语义 |
| 复杂度 | ⭐⭐⭐ | 需要协调规则和大模型 |
| 性能 | 中等 | 部分规则+部分API调用 |
| 成本 | 中等 | 减少大模型调用 |

**优点**：
- 规则部分100%准确
- 大模型专注语义理解
- 可调试性好

**缺点**：
- 实现复杂度增加
- 需要维护规则

### 方案4：分步大模型匹配

**原理**：分多步调用大模型，每步专注一个任务。

```java
public MatchResult calculateStepByStep(User user, Job job) {
    // Step 1: 技能提取与归一化
    String step1Prompt = String.format("""
        从以下文本中提取技能，并归一化到标准标签：

        文本：%s

        返回JSON数组格式：
        {"skills": ["Java", "Spring", "MySQL"]}
        """, user.getRawResume());

    List<String> userSkills = llm.extractSkills(step1Prompt);
    List<String> jobSkills = llm.extractSkills(job.getDescription());

    // Step 2: 技能交集计算
    List<String> matched = intersection(userSkills, jobSkills);
    double skillScore = matched.size() * 100.0 / jobSkills.size();

    // Step 3: 生成匹配理由
    String reasonPrompt = String.format("""
        根据以下匹配信息，生成匹配理由：

        匹配技能：%s
        用户技能：%s
        职位技能：%s

        返回JSON：
        {"reasons": ["原因1", "原因2"], "gaps": ["差距1"]}
        """, matched, userSkills, jobSkills);

    MatchReason reason = llm.generateReason(reasonPrompt);

    return new MatchResult(skillScore, reason);
}
```

| 指标 | 评分 | 说明 |
|------|------|------|
| 精准度 | 85% | 每步专注单一任务 |
| 复杂度 | ⭐⭐ | 需要多次API调用 |
| 性能 | 中等 | 多次网络调用 |
| 成本 | 中等 | 按调用次数计费 |

**优点**：
- 每步可独立调试
- 结果可解释性强
- 方便定位问题

**缺点**：
- 多次API调用，延迟高
- 流程复杂

### 方案5：多阶段匹配

**原理**：初筛（规则快速过滤）+ 精筛（大模型详细匹配）。

```java
public MatchResult calculateMultiStage(User user, Job job) {
    // Stage 1: 规则初筛（毫秒级）
    if (!passBasicFilter(user, job)) {
        return MatchResult.notMatch(); // 直接排除
    }

    // Stage 2: 大模型精筛
    String prompt = buildMatchPrompt(user, job);
    return llm.calculateMatch(prompt);
}

private boolean passBasicFilter(User user, Job job) {
    // 城市必须匹配（硬性条件）
    if (!user.getTargetCity().equals(job.getCity())) {
        return false;
    }

    // 薪资必须重叠
    if (!salaryOverlap(user.getExpectedSalary(), job.getSalaryRange())) {
        return false;
    }

    // 学历必须满足
    if (!educationSatisfied(user.getEducation(), job.getEducationRequired())) {
        return false;
    }

    return true;
}
```

| 指标 | 评分 | 说明 |
|------|------|------|
| 精准度 | 85% | 初筛保证基本条件 |
| 复杂度 | ⭐⭐⭐⭐ | 需要设计初筛规则 |
| 性能 | 好 | 初筛快，节省API调用 |
| 成本 | 中等 | 减少无效API调用 |

**优点**：
- 性能优化，减少API调用
- 硬性条件过滤

**缺点**：
- 可能过滤掉潜在匹配
- 需要维护初筛规则

### 方案8：图数据库匹配

**原理**：利用Neo4j的图关系进行多跳匹配。

```cypher
// 查找与目标职位技能匹配的用户
MATCH (j:Job {id: $jobId})-[:REQUIRES]->(skill:Skill)
MATCH (u:User)-[:HAS_SKILL]->(userSkill:Skill)
WHERE gds.similarity.cosine(skill.embedding, userSkill.embedding) > 0.7
RETURN u, collect(skill) as matchedSkills
ORDER BY size(matchedSkills) DESC
LIMIT 20
```

```java
public MatchResult calculateByGraph(User user, Job job) {
    // 技能关系扩展
    String cypher = """
        MATCH (u:User {id: %d})
        MATCH (j:Job {id: %d})
        MATCH (u)-[:HAS_SKILL]->(userSkill)
        MATCH (j)-[:REQUIRES|SKILL_PREFERRED]->(jobSkill)
        OPTIONAL MATCH (userSkill)-[:RELATED_TO|SIMILAR*1..2]->(jobSkill)
        RETURN u, j,
               collect(DISTINCT userSkill) as userSkills,
               collect(DISTINCT jobSkill) as jobSkills,
               count(DISTINCT relationship(userSkill, jobSkill)) as relationCount
        """;

    GraphResult result = neo4j.execute(cypher, user.getId(), job.getId());

    return calculateMatchFromGraph(result);
}
```

### 方案9：协同过滤匹配

**原理**：基于"看过该职位的还看过"的用户行为进行推荐。

```java
public MatchResult calculateByCollaboration(User user, Job job) {
    // 1. 查找看过该职位的用户
    List<User> similarUsers = viewHistory.findUsersWhoViewed(job);

    // 2. 查找这些用户还投递过的职位
    Set<Job> recommendedJobs = similarUsers.stream()
        .flatMap(u -> applicationHistory.findAppliedJobs(u))
        .collect(toSet());

    // 3. 排除已投递
    recommendedJobs.removeAll(user.getAppliedJobs());

    // 4. 返回高匹配职位
    return recommendedJobs.stream()
        .sorted((j1, j2) -> compareByScore(j1, j2))
        .findFirst();
}
```

### 方案10：技能层级匹配

**原理**：考虑技能之间的父子关系（如Spring Boot → Spring）。

```java
public MatchResult calculateBySkillHierarchy(User user, Job job) {
    // 技能层级树
    // Java
    //   └── JavaEE
    //         └── Spring
    //               ├── Spring Boot
    //               └── Spring Cloud

    Set<String> normalizedSkills = normalizeByHierarchy(user.getSkills());

    // 计算匹配时，父技能匹配则子技能也匹配
    Set<String> matched = new HashSet<>();
    for (String userSkill : normalizedSkills) {
        for (String reqSkill : job.getRequiredSkills()) {
            if (isParentOrEqual(userSkill, reqSkill, skillTree)) {
                matched.add(reqSkill);
            }
        }
    }

    double score = matched.size() * 100.0 / job.getRequiredSkills().size();
    return new MatchResult(score);
}

private boolean isParentOrEqual(String skill1, String skill2, SkillTree tree) {
    if (skill1.equals(skill2)) return true;
    SkillNode node = tree.find(skill1);
    return node != null && node.isAncestorOf(skill2);
}
```

| 指标 | 评分 | 说明 |
|------|------|------|
| 精准度 | 70% | 层级关系固定，不够灵活 |
| 复杂度 | ⭐⭐⭐ | 需维护技能层级树 |
| 性能 | 好 | 树遍历O(log n) |
| 成本 | 低 | 无额外费用 |

**优点**：
- 解决技能粒度问题
- 父子技能自动匹配

**缺点**：
- 技能分类争议大
- 维护成本高

## 场景推荐

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| **快速上线** | 方案2：纯大模型匹配 | 实现最简单，效果好 |
| **追求精准** | 方案2：纯大模型匹配 或 方案7：向量检索+大模型 | 大模型语义理解能力强 |
| **量大性能优化** | 方案6：批量匹配优化 或 方案5：多阶段匹配 | 减少API调用次数 |
| **成本敏感** | 方案3：大模型+规则混合 或 方案1：纯规则匹配（不推荐） | 规则保证精确，大模型减少调用 |
| **海量数据检索** | 方案7：向量检索+大模型 | 向量检索快，大模型精细 |

## 最终推荐

### 首选：方案2 - 纯大模型匹配

```json
// 最佳实践的prompt
{
  "system": "你是一个专业的HR，擅长评估候选人与职位的匹配度。评分要客观公正。",
  "user": "候选人信息：...\n职位信息：...\n请返回详细的匹配分析。"
}
```

**理由**：
1. 实现最简单，代码量最少
2. 效果最好，精准度最高
3. 可解释性强，生成匹配理由
4. 后期优化空间大

### 备选：方案3 - 大模型+规则混合

当API调用量大或需要更精细控制时使用。
