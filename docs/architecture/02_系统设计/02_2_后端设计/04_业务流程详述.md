# 业务流程详述

## 6.1 简历上传解析流程

```
1. 用户选择本地DOC/PDF文件
2. 前端校验文件类型和大小（≤10MB）
3. 调用 /person/resume/upload 上传
4. 后端：
   a. 保存文件到RustFS
   b. 创建resume记录（parse_status=0）
   c. 创建parse_task任务（status=0）
   d. 将任务推送到RocketMQ队列
5. AI服务消费任务：
   a. 更新任务状态为"执行中"
   b. 调用Apache Tika解析文档
   c. 调用NER提取实体
   d. 调用大模型生成技能标签
   e. 更新resume.parse_result
   f. 更新resume.parse_status=2
   g. 触发图谱构建
5. 用户可轮询 /person/resume/{id}/detail 查询状态（解析完成后会收到站内通知）
```

## 6.2 职位发布解析流程

```
完整流程：企业上传职位 → 文件存储 → 消息队列 → 文档解析 → DeepSeek提取 → 存入数据库 → 构建图谱 → 进入匹配池

1. 企业用户填写职位基本信息
2. 可选上传职位描述DOC/PDF
3. 调用 /company/job 新增职位
4. 后端：
   a. 保存文件到RustFS（如有上传）
   b. 创建parse_task任务
   c. 将任务推送到RocketMQ队列
5. RocketMQ消费者接收任务
6. AI服务执行解析：
   a. 从RustFS读取文件
   b. Apache Tika解析文档 → 提取纯文本
   c. 文本 + 提取prompt → DeepSeek API
   d. DeepSeek返回结构化JSON
   e. 解析结果存入job.parse_result
   f. 提取技能存入job_skill表
7. 图谱构建（Neo4j，可选）
8. 管理员审核职位内容
9. 职位上架后进入匹配池
```

## 6.3 双向匹配流程

```
触发时机：
- 个人用户上传/更新简历后
- 企业职位上架/更新后
- 用户主动请求推荐

匹配计算（AI大模型）：
1. 查询用户的技能列表、意向条件
2. 查询职位的技能要求、工作条件
3. 将数据发送给DeepSeek API
4. DeepSeek计算多维度匹配得分
5. DeepSeek生成匹配报告（JSON）
6. 存入match_record表
7. 通过WebSocket/轮询通知用户
```

### 大模型匹配计算

```json
// 发送给DeepSeek

```

## 6.4 图谱构建流程（Neo4j）

```
简历解析完成触发：
1. 读取resume.parse_result中的技能列表
2. 在Neo4j中查找或创建技能节点
3. 创建/更新"人-技能"关系
4. 设置关系属性：熟练度、来源、时长

职位解析完成触发：
1. 读取job.parse_result中的技能要求
2. 在Neo4j中查找或创建技能节点
3. 创建/更新"职位-技能"关系
4. 设置关系属性：是否必须、权重

图谱更新：
- 简历更新 → 重建相关图谱关系
- 职位更新 → 重建相关图谱关系
```

## 6.5 通知消息流程

```
通知触发场景与流程：

1. 简历解析完成
   - AI服务解析完成 → 创建notification(type=1) → 通知用户

2. 新职位推荐
   - 匹配计算完成后 → 筛选高匹配度职位 → 创建notification(type=2) → 通知用户

3. 收到候选人推荐
   - 匹配计算完成后 → 筛选高匹配度候选人 → 创建notification(type=3) → 通知企业

4. 企业认证结果
   - 管理员审核完成 → 创建notification(type=4) → 通知企业

5. 简历被查看
   - 企业用户查看匹配详情 → 创建notification(type=5) → 通知简历所属用户

通知渠道：
- 站内通知：存入notification表，用户通过/notification/list接口查询
- 邮件通知：关键事件（如认证结果）发送至注册邮箱
```

### 6.5.1 通知触发点与API入口对照

| 通知类型 | 触发API | 触发时机 | 创建入口 |
|----------|---------|----------|----------|
| 1-简历解析完成 | `POST /person/resume/upload` | parse_status 变为 2 | AI解析服务 |
| 2-新职位推荐 | `POST /person/match/recommend` | 匹配计算完成 | 匹配服务 |
| 3-收到候选人推荐 | `POST /company/match/recommend` | 匹配计算完成 | 匹配服务 |
| 4-企业认证结果 | `PUT /admin/company/{id}/auth` | 管理员审核完成 | Admin服务 |
| 5-简历被查看 | `GET /company/match/{resumeId}/detail` | 企业用户查看简历详情 | 匹配服务 |

### 6.5.2 简历被查看通知的限制

- **触发条件**：企业用户（user_type=2）查看匹配详情时
- **同企业去重**：同一企业内多个员工查看同一简历，仅在**首次查看**时发送通知
- **跨企业独立**：A企业的查看行为不会通知到B企业相关的候选人

## 6.6 企业认证审核流程

```
企业注册 → 管理员审核 → 认证通过/拒绝 → 通知结果

1. 企业用户注册
   a. 填写企业信息（名称、统一社会信用代码）
   b. 创建管理员账号（sys_user + company_staff.post=OWNER）
   c. 上传营业执照（license_path）
   d. auth_status = 0（待审核）
   e. 注册完成，提示"等待管理员审核"

2. 管理员审核
   a. 管理员登录后台
   b. 查看待审核企业列表 GET /admin/company/list?auth_status=0
   c. 查看企业详情 GET /admin/company/{id}/detail
   d. 审核操作 PUT /admin/company/{id}/auth
      - 通过：auth_status = 1，职位可上架
      - 拒绝：auth_status = 2，记录拒绝原因

3. 通知企业
   - 审核完成后创建 notification(type=4)
   - 通知内容包含：审核结果、拒绝原因（如有）
```

## 6.7 企业员工账号管理流程

```
企业主创建员工 → 员工登录 → 各司其职

1. 员工账号创建（仅企业主/管理员可操作）
   a. 企业主登录 POST /company/staff/create
   b. 填写员工信息（用户名、密码、职务）
   c. 创建 sys_user(user_type=2) + company_staff(post=HR/RECRUITER)
   d. 新员工通过自己账号登录

2. 职务权限说明

   | 职务 | 可操作范围 |
   |------|-----------|
   | OWNER（企业主） | 企业信息管理、员工管理、职位管理（发布/编辑/上架/下架）、查看所有候选人 |
   | HR（HR专员） | 职位管理（发布/编辑/上架/下架）、查看所有候选人 |
   | RECRUITER（招聘专员） | 查看分配给自己的候选人、查看自己发布的职位 |

3. 员工登录
   - 员工用自己的 username + password 登录
   - 登录后 company_staff 表关联员工所属企业
   - 所有操作自动携带 company_id 标识
```

## 6.8 数据权限控制

### 6.8.1 企业用户数据隔离

```
企业A的数据 ←→ 企业B的数据 完全隔离

1. 职位数据隔离
   - 企业用户只能操作自己公司的职位
   - API层校验：job.company_id = 当前用户的 company_id
   - 跨企业访问 → 返回 403 Forbidden

2. 候选人数据隔离
   - 企业只能查看匹配到自己职位的候选人
   - match_record 通过 job.company_id 关联到企业
   - 企业无法查看其他企业的候选人信息

3. 匹配记录隔离
   - 个人用户：只能查看自己的 match_record（按 resume.user_id）
   - 企业用户：只能查看自己企业的 match_record（按 job.company_id）
```

### 6.8.2 管理员数据权限

```
管理员可查看和操作所有数据，但受限于业务规则

1. 用户管理
   - 可查看/启用/禁用任意用户
   - 不可查看用户密码（BCrypt加密不可逆）

2. 企业认证
   - 仅可审核状态为"待审核"的企业
   - 审核操作不可逆（拒绝后可重新提交）

3. 数据维护
   - 可查看所有简历/职位/匹配记录
   - 删除操作需谨慎（级联影响匹配数据）
```

### 6.8.3 权限校验示例

```
# 简历查看权限校验
GET /person/resume/{id}

if (当前用户.user_type == 1) {
    // 个人用户：只能查看自己的简历
    assert resume.user_id == 当前用户.id
} else if (当前用户.user_type == 2) {
    // 企业用户：通过 company_staff 查找 company_id
    // 只能查看匹配到自己企业职位的简历
    assert match_record.company_id == 当前用户.company_id
} else {
    // 管理员：无限制
}

# 职位查看权限校验
GET /company/job/{id}

if (当前用户.user_type == 2) {
    // 企业用户：只能操作自己企业的职位
    assert job.company_id == 当前用户.company_id
}
```
