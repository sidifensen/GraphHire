## 7.3 DDD各层职责

```
┌─────────────────────────────────────────────────────────────────────┐
│                         Interface Layer（接口层）                      │
│   Controller → DTO(Request/Response) → Converter                    │
│   职责：HTTP协议适配、参数校验、返回格式化                              │
└───────────────────────────────┬─────────────────────────────────────┘
                              │ 调用
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│                        Application Layer（应用层）                    │
│   AppService → Command/Query → EventHandler                          │
│   职责：用例编排、事务管理、事件发布（不包含业务逻辑）                    │
└───────────────────────────────┬─────────────────────────────────────┘
                              │ 调用
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│                          Domain Layer（领域层）                        │
│   Aggregate → Entity → Value Object → Domain Service → Event         │
│   职责：核心业务逻辑、业务规则、领域模型完整性                           │
│   关键原则：领域层不依赖任何外层，完全独立                               │
└───────────────────────────────┬─────────────────────────────────────┘
                              │ 调用接口
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│                     Infrastructure Layer（基础设施层）                 │
│   RepositoryImpl → Mapper → MQ Producer/Consumer → External Client   │
│   职责：技术实现、持久化、第三方服务集成                                 │
└─────────────────────────────────────────────────────────────────────┘
```

## 7.4 关键DDD模式实现

### 7.4.1 聚合根（Aggregate）设计

```java
// 聚合根示例：Resume
@Entity
public class Resume extends AggregateRoot {

    @Id
    private Long id;
    private Long userId;
    private String fileName;
    private String filePath;
    private ParseStatus status;          // 状态由聚合根管理
    private ParseResult parseResult;     // 值对象

    // 聚合根负责维护内部一致性
    public void upload(String filePath) {
        this.filePath = filePath;
        this.status = ParseStatus.PENDING;
        this.registerEvent(new ResumeUploadedEvent(this));  // 发布领域事件
    }

    public void parsed(ParseResult result) {
        this.parseResult = result;
        this.status = ParseStatus.SUCCESS;
        this.registerEvent(new ResumeParsedEvent(this));
    }

    // 不允许外部直接修改状态
    private setStatus(ParseStatus status) { ... }
}
```

### 7.4.4 仓库接口（领域层定义）

```java

public interface ResumeRepository {

    Optional<Resume> findById(Long id);

    List<Resume> findByUserId(Long userId);

    Resume save(Resume resume);

    void delete(Resume resume);

    // 领域特有的查询方法
    Optional<Resume> findDefaultByUserId(Long userId);
}

// 仓库实现在基础设施层
@Repository
public class ResumeRepositoryImpl implements ResumeRepository {

    @Autowired
    private ResumeMapper resumeMapper;

    @Override
    public Resume save(Resume resume) {
        ResumePO po = ResumeConverter.toPO(resume);
        resumeMapper.insert(po);
        return ResumeConverter.toDomain(po);
    }
}
```

### 7.4.5 值对象（Value Object）

```java
// 值对象示例：薪资范围
public class SalaryRange {

    private final Integer min;
    private final Integer max;
    private final String unit;

    private SalaryRange(Integer min, Integer max, String unit) {
        this.min = min;
        this.max = max;
        this.unit = unit;
    }

    public static SalaryRange of(Integer min, Integer max, String unit) {
        if (min > max) {
            throw new IllegalArgumentException("min cannot greater than max");
        }
        return new SalaryRange(min, max, unit);
    }

    // 值对象不可变，仅提供查询方法
    public boolean isInRange(Integer salary) {
        return salary >= min && salary <= max;
    }
}
```

## 7.5 包依赖规则

```
依赖方向规则（Dependency Rule）：
┌─────────────────────────────────────────────┐
│  interface层 ──────► application层 ──────► domain层          │
│       ▲                    ▲                    │             │
│       │                    │                    │             │
│       └────────────────────┴────────────────────┘             │
│                      infrastructure层                          │
└─────────────────────────────────────────────────────────────┘

关键原则：
1. 领域层（domain）是核心，不依赖任何外层
2. 应用层（application）依赖领域层
3. 接口层（interface）依赖应用层和领域层
4. 基础设施层（infrastructure）实现领域层定义的接口
5. 同层之间可以互相依赖，但禁止跨层反向依赖
```

## 7.8 与传统三层架构对比

| 维度 | 传统三层架构 | DDD分层架构 |
|------|-------------|-------------|
| **核心** | 数据表为中心 | 领域模型为中心 |
| **业务逻辑** | Service层 | Domain Service + Aggregate |
| **贫血/富血模型** | 贫血模型（POJO只有getter/setter） | 富血模型（实体包含业务逻辑） |
| **依赖方向** | Controller → Service → Mapper | 依赖方向严格遵守外→内 |
| **模块化** | 按技术分层 | 按业务限界上下文分包 |
| **扩展性** | 难以应对复杂业务 | 易于演进和扩展 |
| **适用场景** | CRUD为主的简单业务 | 复杂业务逻辑、领域模型复杂 |
