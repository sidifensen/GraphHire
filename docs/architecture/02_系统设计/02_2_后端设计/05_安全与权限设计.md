# 安全与权限设计

## 8.1 技术选型：Sa-Token

本项目采用 **Sa-Token** 作为认证与授权框架（版本 1.45.0）。

### Sa-Token 核心特性

| 特性 | 说明 |
|------|------|
| 轻量级 | 核心包仅几十KB，无第三方依赖 |
| 注解驱动 | 通过 `@SaCheckLogin`、`@SaCheckRole` 等注解实现权限校验 |
| 灵活存储 | 支持 Redis、Memory、JDBC 等多种 Token 存储方案 |
| 幂等性 | 提供 Token 抽取、校验、二级验证等完整功能 |

### 本项目存储方案

采用 **Redis 存储 Token**，实现分布式 Session 共享：

```yaml
# application.yml
sa-token:
  token-name: Authorization
  token-prefix: Bearer
  expire-time: 7200        # access_token 有效期：2小时
  active-timeout: -1       # 不设置主动过期
  is-concurrent: true     # 允许并发登录
  is-share: true          # 多端互斥
  token-style: uuid       # Token 格式
```

```yaml
# Redis存储配置
spring:
  data:
    redis:
      host: localhost
      port: 6379
      password: ${REDIS_PASSWORD}
      database: 0
```

## 8.2 Token 机制

### 双 Token 方案

| Token | 有效期 | 存储位置 | 用途 |
|-------|--------|----------|------|
| access_token | 2小时 | Redis | 接口访问认证 |
| refresh_token | 7天 | Redis | 续期 access_token |

### Token 流程

```
用户登录
    ↓
验证用户名密码（BCrypt）
    ↓
生成 access_token（存Redis，key: satoken:login:{userId}）
    ↓
生成 refresh_token（存Redis，key: satoken:refresh:{userId}）
    ↓
返回给前端 { access_token, refresh_token, user_type }
```

### 前端请求头

```
Authorization: Bearer <access_token>
```

### Token 续期逻辑

```java
@PostMapping("/refresh")
public Result<LoginResp> refresh(@RequestHeader("Authorization") String authHeader) {
    String refreshToken = authHeader.replace("Bearer ", "");
    // 验证 refresh_token 有效性
    if (saTokenDao.get(refreshToken) == null) {
        throw new AuthException("Refresh token 已过期");
    }
    // 删除旧 Token
    saTokenManager.getStorage().delete(refreshToken);
    // 生成新 Token
    return login(userId);
}
```

## 8.3 登录机制

### 登录接口

```
POST /api/auth/login
Content-Type: application/json

{
  "username": "user@example.com",
  "password": "xxx",
  "captchaKey": "uuid-xxx",      // 图片验证码 key
  "captchaCode": "a1b2"          // 图片验证码 值
}
```

### 登录响应

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "access_token": "xxxx-xxxx-xxxx",
    "refresh_token": "yyyy-yyyy-yyyy",
    "expires_in": 7200,
    "user_type": 1,
    "user_id": 10001
  }
}
```

### 登录限制

- **密码错误锁定**：连续5次失败，锁定账号15分钟（Redis计数器）
- **验证码校验**：图片验证码存 Redis，5分钟有效期
- **密码加密**：BCrypt，不可逆

### Redis Key 设计

| Key | 说明 | 有效期 |
|-----|------|--------|
| `captcha:{uuid}` | 图片验证码 | 5分钟 |
| `login_fail:{username}` | 登录失败计数 | 15分钟 |
| `satoken:login:{userId}` | 用户 Session | 2小时 |
| `satoken:refresh:{userId}` | Refresh Token | 7天 |
| `email_code:{email}:{type}` | 邮箱验证码 | 15分钟 |

## 8.4 注册机制

### 个人用户注册

```
POST /api/auth/register/person
Content-Type: application/json

{
  "username": "user@example.com",
  "password": "Password123",
  "email": "user@example.com",
  "emailCode": "123456"          // 邮箱验证码
}
```

### 企业用户注册

```
POST /api/auth/register/company
Content-Type: application/json

{
  "username": "admin@company.com",
  "password": "Password123",
  "companyName": "XX科技有限公司",
  "companyCode": "91110000XXXXXXXX",  // 统一社会信用代码
  "email": "admin@company.com",
  "emailCode": "123456"
}
```

### 注册流程

```
提交注册信息
    ↓
验证码校验（邮箱验证码）
    ↓
密码强度校验（8-20位，字母+数字）
    ↓
用户名/邮箱唯一性检查
    ↓
BCrypt 加密密码
    ↓
创建 sys_user（user_type=1 个人 / user_type=2 企业）
    ↓
如果是企业用户：创建 company_staff 记录（post=OWNER）
    ↓
发送欢迎通知
```

### 企业资质审核

企业用户注册后 `auth_status=0`（待审核），管理员审核后更新状态：

| auth_status | 说明 |
|-------------|------|
| 0 | 待审核 |
| 1 | 已认证 |
| 2 | 已拒绝 |

## 8.5 角色管理

采用简化的 `user_type + post` 方案，无需 RBAC 五张表。

### 用户类型（user_type）

| user_type | 角色 | 说明 |
|-----------|------|------|
| 1 | 个人用户 | 可上传简历、查看推荐职位 |
| 2 | 企业用户 | 可发布职位、查看候选人 |
| 3 | 管理员 | 系统全部权限 |

### 企业内部职务（post）

| post | 职务 | 说明 |
|------|------|------|
| OWNER | 企业主 | 拥有企业管理权限 |
| HR | HR专员 | 可发布/编辑职位、查看候选人 |
| RECRUITER | 招聘专员 | 仅可查看候选人 |

### 数据表设计

```sql
-- sys_user 表相关字段
user_type   TINYINT      -- 1-个人 2-企业 3-管理员
post        VARCHAR(20)  -- OWNER/HR/RECRUITER（仅企业用户）
company_id  BIGINT       -- 所属企业ID（仅企业用户）

-- 企业员工关联表
CREATE TABLE company_staff (
    id          BIGINT PRIMARY KEY,
    user_id     BIGINT NOT NULL UNIQUE,
    company_id  BIGINT NOT NULL,
    post        VARCHAR(20) DEFAULT 'HR',
    create_time DATETIME,
    update_time DATETIME
);
```

## 8.6 权限校验

### Sa-Token 注解

| 注解 | 说明 |
|------|------|
| `@SaCheckLogin` | 必须登录 |
| `@SaCheckRole("admin")` | 必须拥有指定角色 |
| `@SaCheckPermission("person:resume:upload")` | 必须拥有指定权限（可选） |

### 自定义注解扩展

```java
/**
 * 用户类型校验注解
 */
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequireUserType {
    int[] value();  // 允许的用户类型数组
}

/**
 * 企业内部职务校验注解
 */
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequirePost {
    String[] value();  // 允许的职务数组
}
```

### 拦截器实现

```java
@Component
public class AuthInterceptor implements HandlerInterceptor {

    @Autowired
    private UserRepository userRepository;

    @Override
    public boolean preHandle(HttpServletRequest request,
                            HttpServletResponse response,
                            Object handler) {
        // 1. 校验登录状态（Sa-Token 自动处理）
        // Sa-Token 会验证 Token 有效性并注入当前用户

        // 2. 校验用户状态
        User user = getCurrentUser();
        if (user.getStatus() == 0) {
            throw new BizException("账号已被禁用");
        }

        // 3. 校验企业认证状态（企业用户）
        if (user.getUserType() == 2 && user.getCompany() != null) {
            if (user.getCompany().getAuthStatus() != 1) {
                throw new BizException("企业未认证");
            }
        }

        return true;
    }
}
```

## 8.7 敏感操作二次验证

对于删除数据等敏感操作，需要二次验证：

```java
/**
 * 二次验证注解
 */
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RequireConfirm {
    String message() default "此操作需要二次确认";
}
```

```java
@DeleteMapping("/job/{id}")
@RequireUserType(2)
@RequireConfirm(message = "确认删除该职位？")
public Result<Void> deleteJob(@PathVariable Long id) {
    // 业务逻辑
}
```

前端在调用删除接口前，先弹出确认框获取用户确认，再携带确认参数请求。
