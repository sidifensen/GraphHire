# 用户端登录注册统一页设计

## 背景与目标
- 当前用户端登录页 (`/login`) 与注册页 (`/register`) 视觉和结构分离。
- 目标是统一为同一套页面框架：地址不同但页面骨架一致，右侧表单区按模式切换（登录/注册）。
- 保留角色切换：登录与注册都支持“求职者/招聘者”双角色。

## 设计范围
- 前端路由与页面结构：`/login`、`/register`。
- 用户端认证页 UI：背景、左侧品牌区、右侧卡片与表单切换。
- 现有认证接口调用逻辑（登录、验证码、注册）与错误提示复用。

## 方案概览
1. 新增共享组件 `AuthUnifiedPage`，承载统一布局和两类表单。
2. `app/login/page.tsx` 与 `app/register/page.tsx` 只做模式入口（默认 `login`/`register`）。
3. 在共享组件内提供“登录/注册”切换按钮，点击时通过 `router.push('/login'|'/register')` 切换地址并切换表单。
4. 保留登录中的“求职者/招聘者”切换和注册中的“我是求职者/我是招聘者”切换。

## 视觉与交互
- 整体风格对齐管理端登录页：
  - 左侧：品牌 logo + 大标题文案 + 背景图覆盖。
  - 右侧：玻璃质感卡片、顶部模式切换、表单区。
- 背景图更新为与管理端相似的商务办公风格图（不同图片资源）。
- 在移动端保持单列：卡片优先展示，左侧文案下移为简化信息。

## 数据与行为
- 登录逻辑沿用 `authApi.login` + `resolveLoginRoleDecision`。
- 注册逻辑沿用 `authApi.personRegister` / `authApi.companyRegister` + 验证码发送。
- `/login?review=pending` 仅在登录模式展示审核提示。
- 模式切换时清空跨模式错误提示，避免“登录错误”泄露到注册表单。

## 异常处理
- 登录失败、注册失败、验证码失败继续展示中文错误文案。
- 招聘者注册命中“审核中”语义时，继续展示成功通知并回跳登录审核提示。

## 测试策略
- 先改测试（RED）：
  - 新增断言验证“同一页面可在右侧从登录切到注册、从注册切到登录”。
  - 验证地址模式初始状态：`/login` 默认登录表单，`/register` 默认注册表单。
- 再实现（GREEN）：
  - 仅实现通过测试所需最小代码。
- 重构（REFACTOR）：
  - 抽取共享组件，减少重复。

## 影响与兼容
- 外部入口地址不变：仍可直接访问 `/login` 与 `/register`。
- 原有跳转链接不需要批量改造，只需确保文案与目的地址仍正确。
- 不涉及后端接口、数据库结构、权限模型变更。
