# 用户端真实数据全量接入与正式接口补齐设计规格

**日期：** 2026-04-21  
**状态：** 已批准  
**范围：** 仅求职者前台页面（`/`、`/login`、`/register`、`/companies`、`/jobs`、`/resume/manage`、`/resume/upload`、`/profile`、`/skill-graph`、`/match/[id]`、`/notifications`）

---

## 1. 背景与目标

当前用户端多个页面仍直接使用页面级硬编码数组或静态展示数据，未做到“页面只走正式接口”。现有部分后端接口已能提供基础领域数据，但对首页聚合、企业浏览、图谱分析、通知中心等页面的字段支持仍不完整，前端因此保留了大量 mock 数据。

本次目标是：以页面交付为导向，优先补齐正式后端接口，再统一替换前端假数据，使求职者前台全部页面主数据都来自真实后端接口；若数据库真实数据不足，允许补充结构化测试/演示数据到数据库，但页面源码中不再保留同类业务 mock 列表。

---

## 2. 范围定义

### 2.1 在范围内页面

1. 公共与浏览页
   - `/`
   - `/companies`
   - `/jobs`
2. 求职者账号与资产页
   - `/login`
   - `/register`
   - `/resume/manage`
   - `/resume/upload`
   - `/profile`
3. 匹配与分析页
   - `/skill-graph`
   - `/match/[id]`
4. 通知页
   - `/notifications`

### 2.2 不在范围内

- 企业端页面（`/enterprise/*`）
- 管理端页面（`/admin/*`）
- 与本次页面无直接关系的后台运营能力重构
- 与本次页面无关的大规模表结构重构

---

## 3. 核心原则

### 3.1 真实数据原则

- 页面主内容必须来自正式后端接口。
- 若数据库缺少页面展示所需数据，可以补充结构化测试/演示数据到数据库。
- 前端页面中不得继续保留与真实业务数据等价的硬编码数组作为主数据源。

### 3.2 后端优先原则

- 当前端页面缺少正式接口支持时，优先补后端接口，不允许前端长期拼接多个非页面化接口来模拟正式数据。
- 对强聚合页面，后端应提供稳定的页面聚合查询接口或清晰的查询响应 DTO。

### 3.3 小幅展示适配原则

- 允许为了适配正式接口返回的数据，进行小幅页面结构调整、字段增减、展示顺序调整或空态替换。
- 不允许改变页面核心用途与信息架构主线。

### 3.4 测试驱动原则

- 后端接口补齐必须先补失败测试，再写实现。
- 前端页面切换数据源时，需同步更新或新增对应测试，覆盖真实接口接入后的加载、空态、异常与成功态。

---

## 4. 页面域拆分与实施顺序

本次工作按 4 个页面域分批推进，每个页面域都遵循统一顺序：

1. 盘点页面假数据与字段需求
2. 核对现有接口与表结构
3. 先补后端测试与正式接口
4. 准备数据库可用数据
5. 前端页面切到真实接口
6. 移除对应 mock
7. 联调与页面验证

### 4.1 首页与公共浏览域

包含：`/`、`/companies`、`/jobs`

目标：
- 首页精选职位、热门企业、搜索入口全部来自正式接口
- 公司列表页使用正式企业浏览接口
- 职位列表页使用正式职位列表与筛选接口
- 允许首页侧栏中的静态“装饰型业务数据块”改为真实统计或真实列表

### 4.2 求职者资产域

包含：`/resume/manage`、`/resume/upload`、`/profile`

目标：
- 简历列表、简历上传结果、解析状态、删除能力走正式接口
- 个人信息页走正式查询/更新接口
- 页面状态（解析中、成功、失败、空数据）由真实后端状态驱动

### 4.3 匹配与图谱域

包含：`/skill-graph`、`/match/[id]`

目标：
- 匹配详情页走正式匹配详情接口
- 能力图谱页走正式图谱分析接口
- 后端返回图谱节点、边、维度分析、建议文案或可推导字段
- 允许前端小幅调整展示，以适配后端正式结构

### 4.4 通知域

包含：`/notifications`

目标：
- 通知列表、分类、已读未读、全部已读、跳转目标走正式接口
- 页面分类与角标以正式通知类型映射为准
- 页面不再内嵌示例通知正文

## 5. 后端设计

### 5.1 接口设计原则

后端以“页面可交付”为目标补齐接口，优先在现有模块边界内扩展页面查询能力：

- 公共浏览类能力放在 `publicapi`
- 个人资料、简历相关能力沿用 `resume` / `auth` / `person`
- 匹配与能力图谱能力沿用 `match`
- 通知能力沿用 `notification`

当页面需要聚合数据时，后端应提供明确的查询响应对象，而不是直接把领域实体原样暴露给前端承担拼装责任。

### 5.2 页面域对应接口方向

#### 5.2.1 首页与公共浏览域

需要补齐或增强的正式接口包括但不限于：
- 首页聚合接口：返回首页精选职位、热门企业、热门标签/搜索建议等
- 公开职位列表接口：支持关键词、城市、薪资、技能、排序、分页
- 公开企业列表接口：支持关键词、分页，并根据正式数据补充企业页所需展示字段
- 必要时提供职位/企业详情补充字段，以保证页面卡片信息完整

#### 5.2.2 求职者资产域

需要补齐或增强的正式接口包括但不限于：
- 当前求职者资料查询接口
- 求职者资料更新接口
- 简历列表接口
- 简历上传与上传结果查询接口
- 简历删除接口
- 简历解析状态接口（若页面需要主动轮询或状态刷新）

#### 5.2.3 匹配与图谱域

需要补齐或增强的正式接口包括但不限于：
- 匹配详情接口：返回综合得分、岗位信息、技能重合、差距项、建议文案等
- 求职者推荐职位或匹配摘要接口（如首页或职位页需要）
- 能力图谱接口：返回节点、边、节点分组、分值、对比维度、建议等正式结构

#### 5.2.4 通知域

需要补齐或增强的正式接口包括但不限于：
- 当前用户通知列表接口
- 通知分类过滤接口或统一列表筛选参数
- 未读数接口
- 单条已读接口
- 全部已读接口
- 通知跳转目标字段（页面路由或业务对象标识）

### 5.3 响应结构原则

- 返回结构应优先使用前端可稳定消费的 DTO / Response，而不是直接返回内部领域对象。
- 每个页面查询接口应明确包含页面展示所需字段，减少前端二次兜底。
- 响应中应包含空态所需信息，避免前端依赖猜测逻辑。

## 6. 数据准备设计

### 6.1 数据口径

本次“真实数据”指：
- 页面通过正式后端接口读取的数据
- 数据库存储的真实可读取业务数据或结构化测试/演示数据

### 6.2 数据准备策略

根据页面联调需要，补齐以下类型的数据：
- 企业基础资料
- 职位数据
- 求职者个人资料
- 简历与解析状态数据
- 匹配记录与维度分析数据
- 通知数据
- 图谱展示所需节点/边或可推导原始数据

### 6.3 数据模型策略

- 优先复用现有表结构、现有仓储与现有领域模型。
- 仅当现有结构无法支持正式页面需求时，才补最小必要的数据结构或持久化字段。
- 不为了短期前端展示方便引入明显冗余、不可维护的临时模型。

---

## 7. 前端设计

### 7.1 总体改造目标

前端改造重点是删除页面级 mock 数据源，统一切换到真实加载流，而不是重做整套 UI。

### 7.2 前端统一改造要求

每个页面在接入真实接口后应满足：
- 使用正式 API 请求主数据
- 对接口数据进行稳定 mapper 映射
- 具备 loading 状态
- 具备 empty 状态
- 具备 error 状态与必要的重试入口
- 移除页面源码中的同类业务硬编码数组

### 7.3 允许的页面适配

允许的适配包括：
- 卡片字段增减
- 某些统计块改成真实统计或空态
- 分类项根据正式通知类型或图谱维度重映射
- 匹配详情或图谱分析区的字段顺序调整

不允许的适配包括：
- 改变页面核心目标
- 用前端临时制造业务数据冒充真实接口结果

---

## 8. 测试与验收策略

### 8.1 后端测试

必须补充或更新后端测试，覆盖：
- 用户端页面依赖的正式接口
- 分页、筛选、空数据、异常参数、无权限或无数据场景
- 页面聚合响应结构

### 8.2 前端测试

必须补充或更新前端测试，覆盖：
- 页面接入真实接口后的正常渲染
- loading / empty / error 状态
- 关键字段映射与交互行为

### 8.3 浏览器验收

在功能开发完成后，必须通过 `/web-access` skill 进行浏览器验证，至少覆盖：
- 首页与公共浏览域核心页面
- 求职者资产域核心页面
- 匹配与图谱域核心页面
- 通知页

验收以“页面能打开、能加载真实接口数据、无明显异常”为最低标准。

---

## 9. 完成标准

本次任务完成时，必须同时满足：

1. 求职者前台全部页面主数据来自正式后端接口
2. 页面源码中不再保留同类业务 mock 列表作为主数据源
3. 用户端页面缺失的正式后端接口已补齐或增强
4. 数据库中存在可支撑页面浏览与联调的真实可读数据
5. 后端与前端测试覆盖已同步更新
6. 允许的小幅页面结构调整已统一收口
7. 已通过浏览器对相关页面进行真实联调验证

---

## 10. 风险与约束

### 10.1 主要风险

- 现有领域实体字段不足以直接支撑页面展示，导致需要新增查询 DTO
- 某些页面（特别是首页、图谱、通知）属于天然聚合页，若接口边界处理不好，容易再次把拼装责任推回前端
- 数据库存量数据不足，会导致页面虽然接了接口，但长时间处于空态

### 10.2 控制策略

- 后端优先提供页面稳定响应结构
- 数据准备与接口实现同步推进
- 每完成一个页面域，就清理对应 mock，避免新旧逻辑长期共存
- 最终以浏览器联调为准，而不是仅以单元测试通过为准
