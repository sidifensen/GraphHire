# 企业端真实数据接入与关键操作补齐设计规格

**日期：** 2026-04-21  
**状态：** 已批准（自动推进）  
**范围：** 企业端页面（`/enterprise/dashboard`、`/enterprise/jobs`、`/enterprise/recommendations`、`/enterprise/employees`、`/enterprise/notifications`）

---

## 1. 背景与目标

当前企业端多个页面仍直接使用页面级硬编码假数据，虽然前端已有 `companyApi`、`notificationApi` 等接口封装，后端也提供了部分企业侧控制器能力，但页面展示字段与后端返回结构并未完全对齐，导致企业端首页、职位管理、智能推荐、员工管理、通知中心仍未真正依赖正式后端数据。

本次目标是：

1. 企业端核心页面全部移除主业务假数据，改为通过正式接口读取真实数据；
2. 在后端补齐企业端页面所需的最小接口能力，而不是让前端长期拼接临时数据；
3. 在保留现有 UI 风格的前提下，完成“真实数据 + 关键操作”落地；
4. 关键操作至少覆盖职位状态变更、员工创建、员工密码重置、通知已读等企业端常用流程。

## 3. 核心原则

### 3.1 真实数据原则

- 企业端页面主内容必须来自正式后端接口。
- 页面源码中不得继续保留等价业务 mock 列表作为主渲染源。
- 如果数据库数据不足，可补结构化测试/演示数据，但页面仍只读取正式接口。

### 3.2 页面聚合后端优先原则

- 对仪表盘、推荐页、员工页这类天然聚合页面，优先补后端聚合接口或明确的页面查询 DTO。
- 不把统计拼装、字段推导、跨接口关联长期留给前端完成。

### 3.3 最小可交付原则

- 本次以后端最小必要补齐为目标，不做超范围平台化重构。
- 新增接口与 DTO 只覆盖当前企业端页面真实展示与关键操作所需字段。

### 3.4 测试驱动原则

- 所有新增或增强的企业端后端能力先补失败测试，再写实现。
- 前端页面切换正式数据源时，同步更新页面/API 测试，覆盖成功态、空态、异常态与关键操作态。

## 4. 页面设计

### 4.1 企业首页 `/enterprise/dashboard`

目标：
- 顶部欢迎信息保留静态文案，但统计卡片改为真实数据；
- “待处理投递”“新匹配候选人”“在招职位”来自后端 dashboard 聚合接口；
- “近期发布职位”表格改为真实职位数据，字段至少包含职位名、部门、投递数、匹配候选人数、状态、发布时间；
- 趋势图若当前后端尚无稳定时序统计，则允许降级为最近 7 天聚合数据或最近职位活跃分布，不再用硬编码柱状高度伪造业务结果。

### 4.2 职位管理 `/enterprise/jobs`

目标：
- 列表改为真实职位数据；
- 支持按状态筛选、按关键词搜索；
- 页面展示字段与后端统一：职位名、部门、薪资、发布时间、浏览量、投递数、AI 高匹配人数、状态；
- 关键操作至少支持：发布、关闭/下架、重新解析；
- 若“批量修改”缺乏后端支持，本次允许保留禁用态或占位，但不能假装已接通。

### 4.3 推荐简历 `/enterprise/recommendations`

目标：
- 在招职位单选列表来自真实职位数据；
- 候选人推荐列表来自正式推荐接口；
- 推荐卡片至少包含候选人姓名、头像、教育、工作年限、技能标签、匹配分、推荐理由/分析摘要；
- 支持按职位切换推荐结果；
- “详情”动作至少能拉取候选人详情接口并具备跳转或弹层基础；
- “邀请”若缺少正式接口，本次可先保留展示按钮但不伪造成功结果。

### 4.4 员工管理 `/enterprise/employees`

目标：
- 顶部统计卡片来自员工真实数据聚合：总人数、管理员数量、招聘角色数量；
- 员工列表来自正式接口，字段至少包含姓名/用户名、角色、岗位、状态、最近登录时间；
- 支持创建员工与重置密码；
- 若后端暂未支持完整禁用/恢复/删除，本次可将相关按钮降级为不可用或隐藏，但不能保留假成功行为；
- 角色说明区可继续保留静态说明，因为它属于帮助文本，不属于业务假数据。

### 4.5 通知中心 `/enterprise/notifications`

目标：
- 列表使用正式通知接口；
- 分类基于正式通知类型映射，不再使用示例通知列表；
- 支持全部标记已读；
- 支持单条点击后标记已读；
- “清空已读”若后端未提供能力，本次允许先隐藏或禁用，不做假交互。

## 5. 后端设计

### 5.1 现有能力复用

优先复用现有 `CompanyController` 与 `NotificationController`，并在必要时补充企业端页面专用查询响应结构。

可复用或增强的现有能力包括：
- `/company/info`
- `/company/job/list`
- `/company/job/{id}`
- `/company/job/{id}/status`
- `/company/job/{id}/publish`
- `/company/job/{id}/close`
- `/company/job/{id}/parse`
- `/company/recommend/resumes`
- `/company/match/{resumeId}`
- `/company/staff/create`
- `/company/staff/{staffId}/reset-password`
- `/notifications/*`

### 5.2 需要新增或增强的企业端后端能力

#### 5.2.1 企业首页聚合接口

新增企业 dashboard 聚合接口，返回：
- 待处理投递数
- 新匹配候选人数
- 在招职位数
- 近 7 天趋势数据（若可稳定提供）
- 近期职位列表（最多 N 条）

建议路径：`GET /company/dashboard`

#### 5.2.2 职位列表查询增强

增强 `GET /company/job/list`，支持：
- `status`
- `keyword`

并统一返回页面可直接消费的字段：
- `id`
- `title`
- `department`
- `salaryMin`
- `salaryMax`
- `status`
- `viewCount`
- `applyCount`
- `matchCount`
- `createdAt`

必要时以 DTO 替代直接暴露领域对象。

#### 5.2.3 推荐简历查询增强

增强推荐接口，使其支持按职位过滤：
- `GET /company/recommend/resumes?jobId=...`

返回字段至少包含：
- `resumeId`
- `jobId`
- `name`
- `avatar`
- `education`
- `workYears`
- `skillTags`
- `overallScore`
- `matchAnalysis`
- `missingSkills`

#### 5.2.4 员工列表与统计接口

当前后端已有创建员工和重置密码，但缺员工查询接口。需要补齐：
- `GET /company/staff/list`：返回当前企业员工列表
- `GET /company/staff/stats`：返回员工页顶部统计

若复用已有仓储可直接按当前登录用户所属公司查询，则无需额外传 companyId。

#### 5.2.5 通知接口鉴权封装

现有通知接口要求前端传 `userId`。企业端页面不应由前端自行拼接用户 ID 再访问任意用户数据，因此需提供当前登录用户视角的安全封装，至少包括：
- `GET /notifications/me`
- `GET /notifications/me/unread`
- `GET /notifications/me/type/{type}`
- `GET /notifications/me/unread-count`
- `PUT /notifications/me/read-all`

单条已读仍可沿用 `PUT /notifications/{id}/read`，但后端需校验归属。

### 5.3 响应结构原则

- 页面型接口优先返回 DTO/Response，而不是直接返回领域实体；
- 响应结构要尽量与前端页面展示一一对应，减少前端补字段与猜测逻辑；
- 空数据时返回合法空结构，而不是 `null`；
- 关键操作返回结果应可支持前端即时反馈，如员工重置密码返回新密码。

## 6. 前端设计

### 6.1 总体改造方式

前端改造聚焦于：
- 用正式 API 替换页面级硬编码数组；
- 新增或调整 `companyApi` / `notificationApi` 方法，使其与后端能力一致；
- 页面中统一补 loading / empty / error 状态；
- 关键操作加入按钮 loading、成功提示和失败提示；
- 不重做整体 UI 结构，只做真实数据适配。

### 6.2 页面改造要求

每个企业端页面接入真实数据后应满足：
- 首屏可发起正式请求；
- 数据成功时展示接口内容；
- 无数据时展示明确空态；
- 请求失败时展示错误态与重试入口；
- 页面源码中无同类业务 mock 主数据源残留。

### 6.3 关键操作要求

本次必须打通的关键操作：
- 职位发布/关闭/重新解析；
- 创建员工；
- 重置员工密码；
- 通知全部已读、单条已读。

允许降级但必须显式处理的操作：
- 批量修改职位
- 批量下架职位
- 邀请候选人
- 清空已读通知
- 禁用/恢复/删除员工

这些未接通能力在 UI 上应禁用、隐藏或提示“暂未开放”，不能伪造成功交互。

## 9. 风险与控制

### 9.1 主要风险

- 现有领域对象不完全适合页面直接渲染，需要新增 DTO；
- 推荐接口与职位列表字段可能和前端当前类型不一致；
- 员工信息模型较轻，最近登录时间等字段可能无法完整提供；
- 通知现有接口按 `userId` 暴露，存在前端误用和权限风险。

### 9.2 控制策略

- 采用页面型 DTO，收敛字段；
- 优先补最小后端接口，而不是让前端做长期适配层；
- 对暂不具备的数据字段使用明确空态或弱化展示，不制造假数据；
- 对通知接口补 `me` 视角封装，避免把安全责任交给前端。
