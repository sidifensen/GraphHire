# 投递记录页面设计

**日期**: 2026-04-25
**范围**: 用户端 `投递记录` 页面（列表查看 + 状态筛选 + 分页）

---

## 1. 目标

在前端用户端补全“投递记录”页面，支持用户查看已投递职位，并基于状态筛选和分页浏览，不包含撤回、详情跳转等额外交互。

## 2. 信息架构

- 新增路由页面：`/applications`
- 页面模块：
  - 页面标题与说明
  - 状态筛选标签
  - 投递记录列表
  - 分页控件
  - 空态 / 错误态 / 未登录态

## 3. 数据与状态

### 3.1 数据来源

- 复用 `personApi.getApplications()` 获取投递记录。
- 后端当前接口 `/person/application/list` 返回 `Application[]`（非分页结构）。

### 3.2 前端分页策略

- 采用前端分页：
  - 先获取完整列表
  - 按筛选条件过滤
  - 再按 `pageSize=10` 进行切片
- 分页展示条件：过滤后的总数大于 10 时显示分页。

### 3.3 状态筛选

筛选项：
- 全部
- 待处理（`PENDING`）
- 已查看（`VIEWED`）
- 面试邀请（`INTERVIEW_INVITED`）
- 已接受（`ACCEPTED`）
- 已拒绝（`REJECTED`）
- 已撤回（`WITHDRAWN`）

交互约束：切换筛选条件时重置到第 1 页。

## 4. UI 展示字段

每条投递记录展示：
- 职位名称 `jobTitle`
- 公司名称 `companyName`
- 状态标签 `status`
- 投递时间 `appliedAt`（本地化格式化）
- 匹配分 `matchScore`（为空显示 `-`）

## 5. 路由与导航改动

- 侧边栏“投递记录”链接从 `#` 改为 `/applications`。
- 用户端布局侧边栏显示路由前缀新增 `/applications`。

## 6. 异常与降级

- 未登录：提示“请先登录后查看投递记录”。
- 请求失败：显示错误文案和重试按钮。
- 空数据：显示“暂无投递记录”。

## 7. 测试策略

采用 TDD，先写失败测试后实现：

1. 正常渲染投递记录列表。
2. 状态筛选可正确过滤数据。
3. 分页可切换并展示正确页数据。
4. 未登录时显示提示。
5. 接口失败时显示错误并支持重试。
6. 侧边栏“投递记录”链接正确指向 `/applications`。

## 8. 非目标

- 不实现撤回投递。
- 不实现投递详情页。
- 不新增后端分页接口。
