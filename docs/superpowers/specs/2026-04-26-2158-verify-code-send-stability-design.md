# 验证码发送链路稳定性修复设计

## 背景
注册页点击获取验证码后，接口存在约10秒阻塞；部分场景邮件已送达但接口返回失败，导致前端提示错误并触发重复点击；紧接着再次请求会命中冷却限流，返回“请求过于频繁”。

## 问题根因
1. `MailService` 使用 `CompletableFuture.get(timeout)` 包裹同步发信，超时后主线程判失败，但 SMTP 侧可能已经完成发送。
2. 验证码发送接口同步等待邮件发送，导致前端等待时间过长。
3. 冷却限流在失败场景也生效，用户在感知失败后立刻重试会被拒绝。

## 目标
1. 获取验证码接口快速返回，不阻塞前端交互。
2. 消除“邮件已送达但接口报错”的误判。
3. 冷却窗口内重复点击不再报错（静默成功），降低用户感知噪音。
4. 保留服务端限流保护，避免恶意刷接口。

## 方案
1. 去除 `MailService` 的应用层二次超时控制，改为直接调用 `JavaMailSender`，仅依赖 SMTP 配置超时。
2. 在 `AuthAppService.sendVerifyCode` 内改为后台异步发信，接口立即返回成功。
3. 冷却判定改为“返回是否允许发送”：
   - 60 秒冷却期内：直接返回成功但不重复发信。
   - 小时限额（10次/小时）仍保持强限制。
4. 异步发信失败时清理验证码并回滚限流计数，避免错误状态影响下一次请求。

## 风险与取舍
1. 接口返回成功后邮件仍有可能异步失败；通过日志与回滚机制降低影响。
2. 冷却期静默成功会弱化“重复点击”反馈，但前端已有倒计时，不会造成明显可用性问题。

## 测试策略
1. 单元测试覆盖：
   - 邮件发送异常时接口不抛业务错误。
   - 冷却期内重复调用不报错且仅触发一次发信。
2. 全量验证：前后端编译 + 前后端测试。
3. 浏览器验证：CDP 打开注册页，验证点击“获取验证码”后按钮倒计时与提示行为。
