# 管理端页面直接替换设计说明

**日期：** 2026-04-24
**状态：** Approved（用户已明确要求“直接替换”，按自动模式执行）
**范围：** `frontend` 管理端页面视觉与布局替换，重点处理 Header、Sidebar、登录页

---

## 1. 目标

将 `docs/graphhire-图谱智聘管理端` 的管理端视觉风格与交互结构迁移到 `frontend` 的 `/admin` 页面体系中，在不破坏现有路由与 API 调用的前提下完成“直接替换”。

## 2. 约束

1. 保持现有 Next.js 路由结构与页面路径不变（如 `/admin/dashboard`、`/admin/users`）。
2. 保持现有数据接口调用逻辑不变（`adminApi` 相关调用继续使用）。
3. Header、Sidebar、Login 页面必须统一成参考模板风格。
4. 不能影响非管理端页面（用户端、企业端）。

## 3. 方案

### 3.1 Header

- 统一使用管理端专用 Header（参考 `Topbar.tsx` 的结构与视觉语言）。
- 保留当前退出登录、用户信息展示功能。
- 调整为浅色玻璃拟态 + 固定顶栏 + 操作区（通知/用户下拉）。

### 3.2 Sidebar

- 统一管理端侧边栏品牌区、导航区、底部版本信息区样式。
- 保持当前管理端菜单路由映射与激活态逻辑。
- 侧边栏宽度、配色、激活指示器与参考模板一致。

### 3.3 登录页

- 直接替换管理端登录页视觉布局为参考模板风格。
- 保留现有账号密码提交、错误提示、loading、登录成功跳转逻辑。
- 去除与模板不一致且无业务价值的装饰性行为，优先保持稳定可用。

### 3.4 管理端容器

- `AdminShell` 统一布局边距、背景、内容滚动行为，使所有管理端页面自动获得新 Header + Sidebar。
- `settings` 页面接入 `AdminShell`，避免与其他管理页风格割裂。

## 4. 数据流与风险

- 数据流不改：页面仍由 `adminApi` 拉取与提交。
- 主要风险：样式替换导致测试快照/选择器失效。
- 控制策略：新增/更新前端测试覆盖 Header、Sidebar、Login 核心交互，保证 RED-GREEN。

## 5. 测试策略

1. 先写失败测试：验证登录页文案/登录行为、侧边栏品牌与导航、Header 用户菜单与退出。
2. 再实现页面替换使测试通过。
3. 回归执行：`npm run test:run`、`npm run build`。
4. 最终使用 web-access + CDP 打开 `/admin/login` 与 `/admin/dashboard` 做可视验证。
