# 登录页角色切换动画设计说明

## 背景与目标

用户端登录页当前已具备“求职者/招聘者”角色切换能力，但缺少可感知的切换动效。目标是在不改变业务行为（登录、路由、账号填充逻辑）的前提下，增强角色切换反馈与表单过渡体验。

## 范围

- 页面：`frontend/src/app/login/page.tsx`
- 动画范围：
  - 角色标签（求职者/招聘者）切换动画
  - 表单区域在角色切换时的过渡动画
- 不变更：认证 API、状态管理、路由跳转、文案语义

## 方案

### 1. 角色标签切换动画

使用 Framer Motion 的共享布局能力（`layoutId`）在激活标签上渲染“滑块背景”（indicator）：

- 标签容器保持 `role="tablist"`
- 两个标签按钮保持 `role="tab"` 与 `aria-selected`
- 激活项内渲染一个绝对定位的 `motion.span` 作为背景层，两个标签复用同一个 `layoutId`
- 切换时由 Framer Motion 自动执行位置与尺寸过渡

### 2. 表单区域切换动画

将登录表单外层包裹在 `AnimatePresence` + `motion.div` 中，以 `activeRole` 作为 key：

- 进入：`opacity: 0 -> 1`，`y: 6 -> 0`
- 退出：`opacity: 1 -> 0`，`y: 0 -> -4`
- 过渡时长控制在 0.2s 左右，避免拖慢交互

### 3. 降噪与可访问性

- 保留现有语义与键盘可操作性，不影响 screen reader 的角色识别
- 使用 `useReducedMotion`，在用户偏好减少动画时弱化位移、仅保留淡入淡出或禁用过渡

## 数据与状态流

- 仍以 `activeRole` 作为唯一角色状态源
- `handleRoleSwitch` 保持原行为：切换角色并在开发环境填充对应默认账号
- 动画只消费 `activeRole`，不引入额外业务状态

## 错误处理与回退

- 若 Framer Motion 不可用或动画被关闭，UI 仍保持现有静态切换能力
- 动画失败不影响表单提交和路由跳转

## 测试策略

- 单元/UI测试（Vitest + Testing Library）：
  - 验证默认激活项仍为求职者
  - 验证点击切换后 `aria-selected` 正确更新
  - 新增断言：激活动画指示器存在并随角色切换
  - 新增断言：表单动画容器存在且标识当前角色
- 回归测试：保持登录页既有测试通过

## 验收标准映射

后续 acceptance 文档将覆盖：

- 标签切换时有可检测的激活动画容器
- 表单容器可根据角色切换并保持功能可用
- 可访问性属性不回退
- 动画不破坏登录核心流程
