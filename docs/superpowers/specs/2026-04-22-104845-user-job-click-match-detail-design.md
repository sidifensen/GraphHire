# 用户端职位点击跳转匹配详情设计

- 日期：2026-04-22
- 主题：用户端职位列表点击后跳转职位详情并展示匹配程度
- 方案：A（复用 `/match/{jobId}`）

## 1. 背景与目标
用户在职位列表页点击职位后，需要进入可直接决策的详情视图，看到：
- 职位基础信息（标题、公司、地点、薪资）
- 职位要求（重点是 requiredSkills）
- 用户与职位的匹配程度（总分、匹配等级、已匹配技能、待补足技能）

本次目标是在不新增后端接口、不新增新路由的前提下，复用现有匹配详情页完成能力闭环。

## 2. 范围
### 2.1 In Scope
- 改造用户职位列表页卡片交互：整卡可点击跳转 `/match/{jobId}`。
- 在匹配详情页显式展示“职位要求”区块，渲染 `requiredSkills`。
- 补充前端测试：列表点击跳转、详情页职位要求渲染。

### 2.2 Out of Scope
- 新增 `/jobs/[id]` 路由。
- 重构后端匹配算法或新增聚合接口。
- 大范围 UI 视觉重构。

## 3. 现状
- `frontend/src/app/(user)/jobs/page.tsx` 已能加载职位列表，但卡片当前无跳转行为。
- `frontend/src/app/(user)/match/[id]/page.tsx` 已按 `jobId` 拉取匹配详情、职位详情和图谱分。
- `publicApi.jobs.getById` 可获取 `requiredSkills`。

## 4. 设计方案
### 4.1 路由与交互
- 在职位列表页引入 Next Router。
- 将每个职位卡片改为整卡点击行为：`router.push(`/match/${job.id}`)`。
- 为卡片增加可点击语义与视觉反馈（`cursor-pointer`、键盘可达、聚焦样式）。

### 4.2 详情页展示
- 在匹配详情页新增“职位要求”板块：
  - 数据来源：`job?.requiredSkills`
  - 有数据：按标签展示。
  - 无数据：展示“该职位暂未提供结构化技能要求”。
- 其余匹配展示逻辑复用现有实现：总匹配度、技能对比、匹配原因、已匹配/待补足技能。

### 4.3 数据流
1. 用户进入 `/jobs`，调用 `fetchPublicJobs` 获取职位列表。
2. 用户点击卡片，前端路由跳转到 `/match/{jobId}`。
3. 详情页并行请求：
   - `/person/match/{jobId}`
   - `/public/jobs/{jobId}`
4. 登录用户存在时附加请求：`/match/person/{personId}/job/{jobId}/graph-score`。
5. 页面合并数据并渲染：职位信息 + 职位要求 + 匹配度。

## 5. 异常与降级策略
- 若职位详情请求失败：职位要求区块显示降级文案，不影响匹配模块展示。
- 若图谱分请求失败：技能细项显示兜底文本，整体页面可继续使用。
- 若路由参数非法：沿用当前错误态“无效的职位参数”。

## 6. 测试策略（TDD）
- 先写失败测试，再写生产代码。
- 前端测试用例：
  1. `jobs` 页面点击职位卡片时触发 `router.push('/match/{id}')`。
  2. `match` 详情页在 `requiredSkills` 存在时展示技能标签。
  3. `match` 详情页在 `requiredSkills` 为空时展示降级文案。

## 7. 验收要点
- 用户点击任意职位卡片可进入对应 `/match/{jobId}`。
- 详情页可看到职位要求与用户匹配程度信息。
- 既有加载态/错误态不回归。

## 8. 风险与后续
- `/match/{jobId}` 语义包含“职位详情”是历史折中，后续若引入独立职位详情路由需补迁移方案。
- 若后端后续扩展职位要求字段（如学历/经验），可在同板块按字段并列扩展。
