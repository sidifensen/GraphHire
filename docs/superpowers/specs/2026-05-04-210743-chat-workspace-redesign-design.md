# 聊天页面双栏重设计（用户端+企业端）设计规格

## 背景与目标

当前用户端与企业端聊天页均采用“列表页+详情页”基础结构，桌面端缺少一屏双栏工作台体验，消息展示与输入交互也较为基础。目标是在不改变现有主题色的前提下，参考用户提供页面风格，统一升级双方聊天体验。

## 范围

- 用户端：`/chat` 与 `/chat/[conversationId]`
- 企业端：`/enterprise/chat` 与 `/enterprise/chat/[conversationId]`
- 前端视觉与交互重构，不改后端聊天接口契约

## 非目标

- 不新增后端聊天接口
- 不改消息协议字段定义
- 不引入第三方富文本编辑器或 emoji 库

## 需求拆解

### 1. 布局

- 桌面端：左侧会话列表 + 右侧聊天面板，同一页面并排显示。
- 移动端：保留现有“先列表后详情”路径，不强制同屏双栏。

### 2. 右侧聊天头部

在聊天面板顶部显示两层信息：
- 第一行：岗位负责人 / 公司
- 第二行：岗位 / 薪资 / 地点
- 右侧按钮：查看职位（可跳转对应职位详情）

### 3. 消息区

- 双方消息均显示头像（优先真实头像，缺省回退首字母头像）。
- 时间不在每条消息内显示；改为按天分组，在分组上方显示一次日期分隔。
- 保留文本/图片/简历消息类型展示能力。

### 4. 输入区

- 保持输入框主体。
- 增加表情按钮，点击展开表情弹层，点击表情插入输入框。
- 增加图片按钮，选择本地图片后发送。
- 右侧提供“发送简历”按钮：不需要用户选择，直接发送默认简历（我的第一份简历）。

## 数据策略

- 会话列表：继续使用 `/chat/conversations`。
- 消息列表：继续使用 `/chat/conversations/{id}/messages`。
- 岗位基础信息：
  - 用户端通过 `publicApi.jobs.getById(jobId)` 补全薪资/地点；
  - 企业端优先 `companyApi.getJobDetail(jobId)`，失败时回退会话摘要信息。
- 头像：
  - 当前登录用户头像来自 auth store；
  - 对端头像若无法稳定获取，先使用字母占位头像，避免阻塞主流程。

## 组件与边界

新增共享组件（仅前端）：
- `ChatWorkspace`：主容器，负责响应式布局与会话选择。
- `ConversationSidebar`：左侧会话列表。
- `ChatPanel`：右侧聊天区（头部、消息流、输入区）。
- `EmojiPopover`：轻量表情面板。

页面层仅负责提供角色上下文：
- 用户端页面传入用户态 API 与路由前缀。
- 企业端页面传入企业态 API 与路由前缀。

## 错误处理

- 会话加载失败：显示顶部错误条，不阻断已加载内容。
- 消息发送失败：保留输入内容并展示错误提示。
- 默认简历缺失（用户端）：按钮禁用并提示“暂无简历可发送”。
- 图片上传失败：提示错误并允许重试。

## 测试策略（TDD）

前端测试新增/更新重点：
- 桌面端双栏布局存在性（会话列表与聊天面板同屏）。
- 移动端点击会话后进入详情态。
- 时间分隔按日期显示一次。
- 表情按钮弹层打开与表情插入。
- 发送简历按钮调用默认简历逻辑。

## 验收标准摘要

- 用户端与企业端均实现一致的聊天视觉结构升级。
- 桌面端双栏、移动端单栏路径符合要求。
- 职位头部信息、头像、按天时间分隔、表情/图片/默认简历发送全部可用。

## 风险与缓解

- 风险：企业端难以获取候选人头像。
- 缓解：先统一占位头像策略，后续若补充接口再无缝替换。

- 风险：会话详情页与列表页重复逻辑导致维护成本升高。
- 缓解：通过共享 `ChatWorkspace` 组件收敛逻辑。
