# Backend Full Optimization Design

## 背景
当前后端已具备可用架构，但仍存在数据库基线漂移、热点查询 N+1、异步执行模型分散、事务边界过大、逻辑删除策略不统一、日志热点开销偏高、分页插件配置不明确等问题。

## 目标
1. 修复数据库结构基线与迁移/代码映射不一致问题，保障新环境可重复初始化。
2. 降低匹配与企业侧查询热点中的 N+1 查询成本。
3. 统一异步执行模型，避免临时线程池与默认公共线程池导致的不确定性。
4. 收紧事务边界，减少长事务对连接池与锁资源的占用。
5. 统一逻辑删除策略，避免查询条件遗漏导致的数据可见性问题。
6. 降低高频路径中无效日志与序列化成本。
7. 显式启用分页拦截器并移除不稳定兜底实现。

## 范围
### 包含
- backend 代码与 SQL 基线脚本修复
- 单元/集成测试补充与调整
- 文档与 release notes 更新

### 不包含
- 前端改动
- 业务功能语义大改
- 外部中间件拓扑变更

## 设计

### 1) 数据库基线一致性治理
- 统一 `match_record` 最终结构为压缩分数字段模型：`match_score/skill_score/requirement_score`。
- 更新 `schema.sql`、`init.sql` 与现有迁移脚本语义一致。
- 若需要补差异，新增迁移脚本并补充注释说明。

### 2) Match 与 Company 热点查询优化
- 在匹配列表/推荐场景引入批量预加载：
  - 批量加载 Job、Resume、PersonInfo
  - 批量统计职位匹配数与会话数
- Repository 增加批量接口（按 ID 集合查询、按 jobIds/resumeIds 聚合统计）。
- 保持接口返回结构不变。

### 3) 统一异步执行模型
- 新增 `@Configuration` 异步线程池配置（命名线程、队列容量、拒绝策略）。
- `CompletableFuture.runAsync` 改为显式注入的 `TaskExecutor/Executor`。
- 移除业务内临时 `Executors.newFixedThreadPool`。

### 4) 事务边界收敛
- 对计算密集/IO 密集路径拆分事务：
  - 事务仅包裹必要写入
  - 批量计算与外部调用在事务外执行
- 避免在大循环 + 远程调用场景使用单个大事务。

### 5) 逻辑删除策略统一
- 为关键 PO 统一 `@TableLogic`（job/resume/company 等），并统一 deleted 字段类型映射。
- 清理重复的手写 deleted 过滤逻辑，保留必要显式条件注释。

### 6) 日志与序列化开销优化
- 删除或降级高频 JSON pretty 日志。
- 仅在 debug 时输出摘要，避免生产路径重复序列化大对象。

### 7) 分页插件显式配置
- 新增 MyBatis-Plus `PaginationInnerInterceptor` 配置。
- 删除 `ParseTaskRepositoryImpl` 中“分页插件失效时的全量切片兜底”。

## 测试策略（TDD）
1. 先新增/修改测试，验证上述问题在旧实现下失败（或不满足断言）。
2. 分批改造生产代码直到测试通过。
3. 执行后端验证：`mvn compile`、`mvn test`。

## 风险与回滚
- 风险：批量查询替换可能影响排序/空值处理。
- 对策：保留原有排序语义并补充测试。
- 回滚：按提交粒度回滚单个优化模块。
