# 聊天沟通页 PDF 预览被拦截修复设计

## 背景
用户在沟通界面点击“预览PDF”后，弹窗出现浏览器提示“该内容被屏蔽了，请联系网站所有者以解决此问题”，无法展示简历内容。

## 现状与根因
当前聊天预览弹窗使用 `iframe` 渲染 `blob:` URL，且设置了 `sandbox="allow-downloads"`。在 Chrome/Edge 的内置 PDF 查看器场景下，该沙箱权限不足会导致 PDF 插件/查看器被阻止，最终出现“内容被屏蔽”的空白页。

## 目标
- 修复聊天页 PDF 预览，确保在主流 Chromium 浏览器中可直接打开。
- 保持现有“下载PDF”能力不变。
- 不改动后端接口和权限逻辑。

## 非目标
- 不改造简历管理页预览能力（本次仅修复沟通页）。
- 不调整消息结构、上传流程、鉴权流程。

## 方案对比
### 方案A：移除/放宽 iframe sandbox
- 优点：改动小。
- 缺点：安全边界模糊，且不同浏览器对 PDF 查看器行为仍可能不一致。

### 方案B：改为 `object[type=application/pdf]` 内嵌预览（推荐）
- 优点：绕开 iframe 沙箱与 PDF 插件冲突，兼容性更稳定；可在标签内提供降级下载入口。
- 缺点：需同步更新测试断言。

### 方案C：直接新窗口打开 blob
- 优点：实现简单。
- 缺点：交互退化，且可能触发弹窗拦截，不符合“弹窗内预览”体验。

## 选型
采用方案B：聊天预览弹窗对 PDF 使用 `<object type="application/pdf">`，并提供“无法预览时下载文件”的回退提示；图片预览逻辑保持不变。

## 测试策略
1. 先修改测试，新增/调整断言至 `object` 预览结构，确保旧实现失败（RED）。
2. 实现最小改动通过测试（GREEN）。
3. 执行前端验证：`npm run test:run`、`npm run build`。
4. 使用 CDP 进行一次实际页面操作验证：聊天页点击“预览PDF”可见 PDF 容器与回退链接。

## 风险与回滚
- 风险：个别浏览器对内嵌 PDF 支持差异导致仅触发回退链接。
- 应对：保留下载按钮与回退下载入口，确保功能可达。
- 回滚：仅需回退 `ChatPreviewModal` 与对应测试改动。
